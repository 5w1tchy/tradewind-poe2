# Tradewind — PoE2 price-check overlay

One hotkey on a hovered Path of Exile 2 item → an overlay popup with a price
estimate and live trade listings. Electron + TypeScript + React.

> **Keep this file current.** If a change alters anything described here —
> commands, the process boundary, the `src/core` purity rule, the IPC surface,
> overlay/popup behavior, the input tiers, or any convention/gotcha below —
> update CLAUDE.md in the **same** change. A stale CLAUDE.md is worse than none.

## Commands

```
npm run dev        # electron-vite dev — launches the overlay, watches for the PoE2 window, HMR
npm run build      # electron-vite build → out/
npm start          # electron-vite preview (built app)
npm test           # vitest run — parser/matcher/query/pricing/craft unit tests
npm run fixtures   # re-split fixtures/inbox.txt into per-item test fixtures
npx tsc -b         # typecheck (project refs: tsconfig.node.json + tsconfig.web.json)
```

Browser UI preview without the game or Electron:
`npx vite --config vite.preview.config.ts` then open `/preview.html` (mocks
`window.tradewind` with a canned item — see `src/renderer/src/preview.ts`).

## Layout & the process boundary

```
src/
├─ core/      Pure TypeScript, NO electron imports. Unit-tested. The brain.
│   ├─ parser/    clipboard item text → ParsedItem (splits on "--------" sections)
│   ├─ stats-db/  live-fetched /api/trade2/data → runtime matcher generation (# placeholders → regex)
│   ├─ query/     ParsedItem + filters → PreparedQuery → trade2 search body (prepare.ts, build.ts)
│   ├─ pricing/   currency normalization, outlier rejection, estimate range + confidence
│   ├─ craft/     essence + Liquid Emotion crafting helpers (essences.json / liquids.json generated by scripts/gen-essences.mjs / gen-liquids.mjs)
│   ├─ mod-pool/  chat-link affix/tier reconstruction (pool.json/bases.json from scripts/gen-mod-pool.mjs)
│   ├─ exchange/  currency-exchange pricing: aggregate price → denominations, trend, liquidity (no live book)
│   └─ trade/     SearchOutcome / listing types
├─ main/      Electron main process. Win32 window tracking, global hotkey, clipboard, API gateway.
├─ preload/   contextBridge — exposes window.tradewind (see src/shared/ipc.ts for the surface)
├─ renderer/  React 19 (.tsx + CSS Modules). The popup UI.
└─ shared/    ipc.ts — the typed IPC contract shared by main, preload, renderer
```

**Golden rule: `src/core` stays pure** — no Electron, no DOM, no `window`. It is
imported by both the main process and the renderer and must stay framework- and
runtime-agnostic. All side effects (network, clipboard, windows) live in `main/`.

## The core loop

1. Native keyboard hook sees `Ctrl+D` while PoE2 is focused; the keypress is swallowed.
2. `main/` synthesizes the game's advanced copy (`Ctrl+Alt+C`), polls the clipboard, restores it.
3. `core/parser` → `ParsedItem`; `core/stats-db` matcher maps mod lines to stat IDs; `core/query/prepare` builds a `PreparedQuery`.
4. Main sends `tw:item` (`ItemPayload`) to the renderer, which shows the popup and auto-searches (uniques/gems/white bases) or arms the Search button (rares). **Currency-exchange items** (anything with an `exchangeId`) skip the live search entirely: main attaches a `currency` quote (poe2scout aggregate) to the payload and the renderer shows the chart view — see the exchange gotcha below.
5. Renderer calls `search()` → main runs `POST /search` + `GET /fetch` through the single rate-limit queue; `core/pricing` produces the estimate; results render.

## Conventions & gotchas

- **React, not Vue.** `PLAN.md` predates the migration and still says "Vue 3 + Tailwind" — ignore that; the renderer is React 19 with **CSS Modules** (`*.module.css`). There is **no Tailwind**. Generic class names (`.name`, `.row`, `.tier`) collide across components, which is why styles are scoped modules; only the global theme primitives (`tw-label`, `tw-btn`, `tw-menu`) in `src/renderer/src/styles/theme.css` are plain global class strings.
- **No `<StrictMode>`.** The preload appends IPC listeners with no unsubscribe, so the dev double-invoke would deliver every item twice. `onItem`/`onHide` are subscribed exactly once in `App.tsx`'s mount effect — keep it that way.
- **Mod-origin tags** (issue #54, `core/query/prepare.ts` `originFor` → `PreparedStatFilter.origin`, rendered by `originTag()` in `PriceCheck.tsx`, colors in `PriceCheck.module.css`). A stat row shows a short colored tag *after* the affix badge for a mod with special provenance: **F** fractured, **C** crafted, **D** desecrated, **E** enhanced (anoint/enchant `{ Enhancement }`), **CE** corruption (`{ Corruption Enhancement }`). Origin is derived from the `ParsedMod` flags (`crafted`/`desecrated`/`fractured`) plus generation; a `{ Corruption Enhancement }` header parses as **enhancement-generation with `mod.corrupted` set** (vs a plain anoint), which is what splits CE from E. Synthetic rows (summed `(total)`, resistance pseudos, the hybrid-single convenience rows) carry **no** origin — the in-place hybrid node owns the tag. **Origin only survives an advanced/inventory copy:** a chat link copies a corruption-enhancement as a bare `(enchant)` line with no marker, so it degrades to a plain enchant row with no CE tag — same "don't guess what the clipboard didn't give us" rule as tier reconstruction.
- **Two clipboard copy formats; only chat links need reconstruction** (`core/parser/parse.ts`, `core/mod-pool/`). An **inventory** copy is the *advanced* form: each mod carries a `{ Prefix Modifier "…" (Tier: N) }` header, so the parser reads affix + tier (and the `(min-max)` roll range) straight off the text — exact, no guessing. A **chat-linked** item copies in the *basic* form: bare mod lines, **no headers, no tier, no ranges** (implicits are still tagged `(implicit)` and split out by the parser). For basic copies, `core/mod-pool/modPool.ts` *reconstructs* affix + tier from the rolled value against the bundled `pool.json` (slimmed repoe-fork mods) + `bases.json`: match the stat text, keep brackets the roll fits, read back `generation_type` + the tier index within that base-spawnable family. **`prepare.ts` only ever reconstructs when `mod.generation === 'explicit' && tier === null`** — advanced/inventory mods keep their parsed header values untouched. **Crafted/desecrated mods reuse the regular tier ladder** (a `(crafted)` 26% cast speed is the same T3 a rolled one would be), so they reconstruct like any explicit — don't special-case them out. Correctness rules in the matcher: spawn weights are **ordered, first-matching-tag-wins** (`!`-prefixed tag = weight 0; a naive "any positive tag" test over-counts tiers and is wrong); brackets come from the **display text** not `stats` (leech is stored permyriad, "reduced" is stored negative); a roll whose tiers overlap, or two mods that collide on one **group** (impossible on a real item), yields no tier; and when a tier can't be pinned but the stat has one fixed affix slot, `affixForStat` still badges the **affix only** (P/S, no number). A roll that fits **both** a prefix and a suffix bracket (e.g. Rarity of Items found) is resolved by **affix-slot constraint** — once the confident mods fill one side (rares cap at 3+3, magic at 1+1) the ambiguous one is forced to the open side (`reconstructExplicits` in `prepare.ts`); the rare `+1 prefix/suffix` crafts can exceed the cap, a documented edge. A wrong number never shows — a blank or affix-only beats a guess. ~99% never-wrong on gear; residual blanks are genuine value ambiguities or stats absent from the upstream dump (e.g. the maximum-mana *suffix* family). Re-run `node scripts/gen-mod-pool.mjs` (reads `scratch/mods.min.json` + `base_items.min.json`, else fetches repoe-fork) after PoE2 patches. **Source is repoe-fork, not poe2db** — repoe-fork is structured JSON with the tiers/tags we need; poe2db has no API (HTML only, scraped solely for essences). `gen-mod-pool.mjs` also emits a separate **`jewelEntries`** table into `pool.json` — the `misc`-domain jewel mods (gear is `item`), as text → group(s) + affix with **no** tier (a group-conflict needs only the group). It's kept apart from `entries` so the two domains' same-text mods (e.g. "increased Area of Effect" exists in both) never cross-resolve. `modPool.ts` exposes `groupsForLine(baseType, line, affix?)` (gear) and `jewelGroupsForLine(line, affix?)` (jewel) — loose group lookups for the conflict gates (#72/#78), distinct from the tier-reconstructing `reconstruct*`.
- **IPC channels** (`tw:*`) are defined in `preload/index.ts` and handled in `main/index.ts`; the typed surface is `src/shared/ipc.ts`. Add new calls in all three.
- **Essence advice respects the crafted-mod cap** (issue #24, `core/craft/craftedSlots.ts`, `essencesForItem`, `CraftPane.tsx`). A Perfect or corrupted essence augments a Rare with a guaranteed mod in its **single crafted slot**, so on a Rare that already fills that slot the game blocks it. `craftedSlots(item)` counts `explicits.filter(m => m.crafted)` (a mod that is **both fractured and crafted** still occupies the slot — this is why `modHeader`'s descriptor regex accepts a *stacked* variant prefix like `Fractured Crafted`, not a single keyword) and reads the cap off the rune lines: base 1, **+N per `Can have N additional Crafted Modifier` rune** (Astrid's Creativity, a socketed Soul Core, copies as that line). Main attaches `{ used, cap }` to `ItemPayload.craftedSlots`; `essencesForItem(class, rarity, craftedSlots?)` drops Perfect **and** corrupted essences when `used >= cap`, replacing the list with a plain reason note instead of silently hiding them. Only Rare gear is gated (Magic/Normal don't have the slot yet). The shared `craftedCapNote(crafted, blocked)` (in `craftedSlots.ts`) builds the reason line for both essences and **liquids** (see next). **Each guaranteed mod in `essences.json` also carries `groups` + `affix`** (joined from the repoe-fork `mods.min.json` by `gen-essences.mjs` — match poe2db's display text to a gear mod by an order-independent word multiset, disambiguating by the granted mod's **exact roll bracket first** (the range pins the tier, hence the prefix/suffix *family* for a stat that rolls as both — e.g. a (15-18)% Rarity essence is the suffix `ItemFoundRarityIncrease`, since the prefix family's neighbouring tier is (16-19)), then by the non-rollable candidate; ambiguous text yields the union of groups, special mods like "Mark of the Abyssal Lord" yield none). This is the data layer for the **mod-group-conflict gate**: two mods can't share a group, so a guaranteed-mod craft is blocked when the item already has a mod in the guaranteed mod's group (e.g. Celestial Alloy's `IncreasedMana` mod on a wand that already has max Mana).
- **Essence advice flags mod-group conflicts** (issue #72, `core/craft/conflict.ts`, `core/mod-pool/modPool.ts` `groupsForLine`, `essencesForItem`, `CraftPane.tsx`). Built on the `groups` data above. Two mods can never share a group, so an essence's guaranteed mod is **refused by the game when the item already carries a mod in that group** — both for a Perfect/corrupted essence augmenting a Rare *and* a Greater essence upgrading a Magic item (e.g. Greater Essence of Opulence on a Magic item that already has a Rarity suffix). Wiring: `collectItemMods(parsed, baseType)` (the `itemMods` export of `conflict.ts`) pairs each **explicit** mod's line text with its repoe group(s) via `groupsForLine` (a deliberately loose pool lookup — all candidate groups for the stat text, base-narrowed when known, since a group is intrinsic to the stat and base-independent; empty when the stat isn't in `pool.json`, so it blocks nothing). `groupsForLine` takes the mod's **affix** (from the advanced copy's prefix/suffix header) and filters candidates by it, so a stat that rolls as both a prefix and a suffix under *different* groups (Rarity of Items found) resolves to the right family — a suffix-rarity item mod doesn't cross-block a suffix-rarity essence's prefix-rarity twin, and vice versa. A basic chat-link copy carries no affix, so both groups stay (the conservative "don't guess" default). This is general, not a rarity special-case — many essences grant such affix-split mods. For a **jewel** (`itemClass === 'Jewels'`), `itemMods` resolves groups via `jewelGroupsForLine` instead — jewel mods are a different mod domain with their own group table (see the mod-pool note), so the two never cross-match. This is what feeds the liquid gate (#78). Main attaches that to `ItemPayload.itemMods`; `essencesForItem(class, rarity, craftedSlots?, itemMods?)` sets `EssenceForItem.blockedBy` to the conflicting mod's display text via `conflictingMod(modGroups, itemMods)`. Unlike the #24 cap gate, **blocked essences stay listed** (not hidden) — the renderer greys the row and shows `blocked by "<mod>"` on the right, so the user sees *why*. The gate no-ops without `itemMods` (every essence lists unblocked). The same `conflict.ts` helpers are the shared check the **#51 Alloys** advisor will reuse. Preview: the default Rings item ships `SAMPLE_ITEM_MODS` (a `MaximumManaIncreasePercent` mod) so Perfect Essence of the Mind shows blocked — use `?used=0` to clear the #24 cap block and see the gate on the list.
- **Liquid Emotions are the jewel-crafting sibling of essences** (issue #48, `core/craft/liquids.ts`, `liquids.json` from `scripts/gen-liquids.mjs`, the **Liquids** sub-tab in `CraftPane.tsx`). Applied to a **Rare jewel**, a liquid removes a random modifier and augments it with a guaranteed **crafted** mod — mechanically the corrupted-essence `rareOnly` path, so it reuses the **#24 crafted-cap gating** (a liquid-applied mod copies as a real `{ Crafted … Modifier }`, so `craftedSlots()` already counts it; jewels have no rune lines so the cap stays 1). **The key difference from essences: liquids key off the jewel's *gem base type*, not the item class** — every jewel is `Item Class: Jewels`, so `liquidsForItem(itemClass, baseType, rarity, crafted?)` reads the base type (rares carry it on `prepared.baseTypeFilter.value`, uniques on `prepared.type`): `Time-Lost <gem>` → Time-Lost jewel, bare `Ruby/Sapphire/Emerald/Diamond` → basic jewel; an unknown base (or non-jewel class) returns an empty no-note result. The poe2db table is **gem×affix keyed** (`Ruby Prefix: …`, `Emerald Suffix: …`) and the **target family is read from the behavior sentence** (`…Augments a Rare Basic Jewel…` vs `…Time-Lost Jewel…`), so each liquid stores `mods: [{ gem, affix, text }]`. **Ruby/Sapphire/Emerald are attribute jewels** — each takes only its own column (one deterministic mod). **Diamond is the wildcard jewel**: if the liquid has its own **Diamond** column it gives only that (e.g. Isolation → chaos res only); lacking one, Diamond rolls a *random one of* the attribute variants (e.g. Ire → armour/ES/evasion). So `liquidsForItem` returns the Diamond column when present else every line for Diamond (deduped by affix+text), and only the matching-gem line(s) for the attribute gems. Since a liquid always adds exactly one mod, **`LiquidForItem.mods` is the set of possible outcomes** — >1 entry means "rolls one of" (the renderer tags it), which covers both Diamond and the **Potent** liquids (Ferocity/Contempt/Melancholy) that offer a prefix *or* suffix. Potent liquids are gold-tinted vs base-emotion bronze and sort last. Liquids only apply to **Rare** jewels (they remove-and-augment, they don't upgrade rarity like Greater essences). Re-run `node scripts/gen-liquids.mjs` after PoE2 patches — it scrapes poe2db's `#LiquidItem` pane (the page's other tabs are the out-of-scope 3-liquid amulet-instilling passives) and downloads icons to `assets/liquids/` (use the saved `scratch/Liquid_Emotions.html` for a deterministic re-run). Preview with `?class=Jewels&base=Emerald&rarity=Rare&used=0` (`preview.ts` `base` param). **Liquids also carry the mod-group-conflict gate** (issue #78, the jewel twin of #72): each liquid mod in `liquids.json` carries `groups` (joined by `gen-liquids.mjs` from the **`misc`** domain — where jewel mods live, vs gear's `item` — using the liquid's known prefix/suffix slot to disambiguate), and `liquidsForItem(..., existing?)` sets a **per-outcome** `LiquidMod.blockedBy` via the shared `conflictingMod`. Per-outcome because a "rolls one of" liquid (Diamond wildcard, Potent prefix/suffix) can have some outcomes blocked and others not — the renderer greys only the blocked lines. The jewel's existing mods are group-resolved by `jewelGroupsForLine` (see the mod-pool note). **Time-Lost** "Passive Skills in Radius also grant …" mods are absent from the repoe dump, so they carry no groups and never block (the conservative default) — only basic-jewel liquids are gated.
- **Currency-exchange items price off poe2scout, not the live trade API** (`main/exchangeSnapshot.ts`, `core/exchange/`, `renderer/components/CurrencyView.tsx` + `PriceChart.tsx`). Everything the in-game Currency Exchange (Ange) trades — currency, fragments, runes, essences, lineage support gems, … — carries **one aggregate price**, not a per-roll order book, so a live search is the wrong tool. When `prepareQuery` sets an `exchangeId`, `priceCheck` resolves a `currency` quote from poe2scout's **flat `/{realm}/Leagues/{league}/Items`** snapshot (one ~450KB GET covering every exchange item, cached via `cachedFetchJson` with a short `freshMs` since prices move) and attaches it to `ItemPayload`. The join is exact: **GGG static exchange id == poe2scout `ApiId` == `exchangeId`** (e.g. `rakiatas-flow`). The renderer branches in `PriceCheck` on `payload.currency` to render `CurrencyView` instead of the search/listings UI — **no Search button, no pricing controls, no API call** — showing the price in all three denominations (ex/div/chaos, derived from the snapshot's own `divine`/`chaos` rows) plus a hand-rolled SVG **price + volume chart**. Chart history (`PriceLogs`, 6-hourly) is fetched lazily per item via `tw:currency-history`; switching league re-resolves via `tw:currency-quote`. **It's a soft dependency** — a snapshot miss/failure leaves `currency` null and the item falls back to the existing live exchange path, so poe2scout being down degrades, never breaks. (This is the same source as the optional `scoutAnchor` cross-check, here promoted to primary for exchange items only — uniques/rares still price live.)
- **Uniques get an instant poe2scout aggregate banner + a manual live search** (issue #80, `main/exchangeSnapshot.ts` `uniqueQuote`, `core/exchange` `UniqueQuote`, `renderer/components/UniqueQuoteBanner.tsx`). The same flat `/Items` snapshot that prices currency **also carries every unique** (466 of them, each with `Name`/`Type`/`CurrentPrice`/`IconUrl`/`ItemId` but **`ApiId: null`**), so a rough ballpark is free at check time. The provider builds a **second index keyed by `name|type`** (uniques have no ApiId) alongside `byApiId`; `uniqueQuote(league, name, type)` joins on parsed `prepared.name` + `prepared.type` (normalized case/space-insensitively). **Unlike currency, this is a soft *anchor*, not a replacement** — poe2scout's aggregate is noisy for cheap/thinly-traded uniques (a 1-ex leveling unique can report ~97k ex at quantity 1–3), so a unique still prices **live, per-roll**: the banner is the instant ballpark, the search below is the truth. Two behavior changes for `rarity === 'Unique'`, both in `PriceCheck.tsx`: (1) it's **excluded from the auto-search gate** (the `name || type || exchangeId` branch) so uniques **arm the Search button** like rares instead of auto-firing a rate-limited search on every hover — the whole point is cutting GGG `search` traffic; (2) `UniqueQuoteBanner` renders on top of the Price tab — the price in ex/div/chaos with currency-orb icons (from `payload.currencyIcons`, same source as `CurrencyView`) + the item art + an "<rarity> ◆ poe2scout aggregate" eyebrow. It's styled as a *sliver of the item's own rarity tooltip*: a single inline `--accent` CSS var (set from the rarity color, `RARITY_ACCENT`) is `color-mix`ed through the spine/wash/border/label so the banner reads as "unique" at a glance, while the price number itself stays **gold** (value is gold — the overlay-wide rule). **Non-blocking by design:** the banner requests its quote on mount via the new `tw:unique-quote` IPC (`getUniqueQuote`, mirrors `getCurrencyQuote`) — main's `priceCheck` hot path is **unchanged** (no quote attached to `ItemPayload`), so the popup is instant even on a cold snapshot. A miss/failure → no banner, Search still works (same soft-dependency posture as currency). Preview with `?rarity=Unique` (`preview.ts` seeds `SAMPLE_UNIQUE`). Out of scope (follow-up): a lazy price-history sparkline via `/Items/{ItemId}/History` and a `Quantity`-based liquidity hint.
- **Popup is a fixed-size, resizable box** (`App.tsx`, issue #35): the popup `<div>` has an explicit width/height (not content-sized), placed **top-middle** (horizontally centered, ~8% down) and **draggable** from the header bar. Because the frame size is fixed, **content changes never resize it** — switching Price↔Craft or loading results leaves the frame put; each tab panel fills the frame via the flex-column `.body` and **scrolls internally** (the stats list, the results list, the essence list own their own overflow). Drag the **bottom-right corner handle** to resize the whole popup; on the Price tab, drag the **grip above the results list** to set its height while the stats list flex-fills the rest. Both sizes persist to `config.json` (`popupSize` via `tw:set-popup-size`, `resultsHeight` via `tw:set-results-height`) and are restored on the next check (sent back in `ItemPayload`). Anchor resize gestures to the rendered `size`/`resultsHeight` state, **not** `el.offsetWidth` (which includes padding+border and snaps the first drag tick). `config.ts`'s `sanitize()` coerces/clamps every loaded field, so a malformed or schema-changed `config.json` heals to defaults instead of leaking `NaN`. The **buyout-price currency** is persisted the same way (issue #20): picking one in the buyout popover fires `setBuyoutCurrency` → `tw:set-buyout-currency`, main stores it as `config.buyoutCurrency`, and `prepareQuery` seeds it into every new check's `buyout.option` (via `PrepareOptions.buyoutOption`) so the picker remembers the choice — `null` = Exalted Orb Equivalent, the only value that adds no price filter; any other option alone restricts listings to that currency.
- **Overlay popup behavior** (`App.tsx` + `main/index.ts`): the popup mimics PoE2's item tooltip (issue #32). The top-right header button is one **morphing control** (`setPinned` → `tw:set-pinned`): a **pin** glyph while unpinned (click to pin), an **✕** while pinned (click to close — there is no separate unpin). **Unpinned** (default): an outside click or Esc closes it. **Pinned**: outside clicks are ignored; only the ✕ or Esc close it, and a **fresh price check updates the popup in place and keeps the pin** — pin state persists across searches and resets to unpinned **only when the popup closes** (`hidePopup` for Esc/outside; the null `tw:popup-rect` for the ✕). The renderer reports its rect via `setPopupRect`; the main process hit-tests the cursor against it to toggle click-through (`setIgnoreMouseEvents`) and, on a `mousedown` outside it (observed via uiohook), auto-hides when unpinned. Report the rect synchronously from placement; never from a stale `pos` state (a `{0,0}` ghost report makes main treat the popup as off in the corner).
- **Software rendering on purpose.** `main/index.ts` calls `app.disableHardwareAcceleration()` at startup. On machines where the GPU is present but only half-engaged for Chromium (it shows up yet `app.getGPUFeatureStatus().gpu_compositing` is a `disabled_software`/WARP fallback), letting Electron try the GPU and limp along in that path makes the popup lag hard (jerky drag, multi-second hover repaints). The overlay is a tiny, mostly static UI that doesn't need the GPU, so forcing clean software rendering is smooth everywhere and dodges that whole fallback class. Don't "optimize" this away.
- **Input has three tiers** (`main/index.ts`, `keyhook.ts`, `input.ts`): native `WH_KEYBOARD_LL` addon (`tradewind-keyhook` via koffi, best — blocks the key before the game's key-state table) → `globalShortcut` → `uiohook-napi` observe-only. The native tier prevents `Ctrl+D` leaking as a WASD-style game input.
- **Tests are fixture-driven.** `fixtures/` holds real in-game `Ctrl+Alt+C` copies. ⚠️ `npm run fixtures` (`scripts/split-fixtures.mjs`) **wipes any manual fixture not present in `fixtures/inbox.txt`** — add new samples to `inbox.txt`, don't hand-place fixture files.
- **Tray is the only chrome.** The sole window is the click-through overlay, hidden whenever PoE2 isn't focused — so the system-tray icon (`main/tray.ts`) is the only way to quit (Quit) and to trigger a manual update check (Check for updates…, wired to `checkForUpdatesManually` in `updater.ts`, which opens a **themed dialog**, not the native grey box: `main/themed-dialog.ts`'s `showThemedDialog` draws the same kind of frameless/transparent/always-on-top window as the tray menu (centered, draggable card, `tw://btn/<i>` nav channel) but is **stateful** — it returns a `DialogController` whose `setContent()` morphs the live window in place, so the manual check walks one window through Checking… → Available (+ Update button) → Downloading… → relaunch. Buttons are callback-based (`onClick`/`keepOpen`); Esc/Enter map to the `cancel`/`primary` buttons). Its menu is **not** the native Win32 menu (`tray.setContextMenu`), which can't be themed: `main/tray-menu.ts` draws a **custom themed popup** (`showTrayMenu`) matching the overlay — a frameless/transparent/always-on-top `BrowserWindow` (same recipe as `splash.ts`, inline-HTML data URL, palette mirrored from `theme.css`), shown at the cursor on **either** tray click (left included, since the native menu's right-click-only is unintuitive for a one-icon app). It owns no preload/IPC: rows navigate to a `tw://item/<i>` scheme that `will-navigate` intercepts; Esc or blur (click-away) dismisses. Keep the `tray` reference alive in `index.ts` or the GC reaps it from the notification area. The app/installer icon is `build/icon.png` (the compass logo, already cropped tight to its content), picked up automatically by electron-builder. `build/` isn't bundled into the asar, so the tray can't read it at runtime — `scripts/gen-tray.mjs` (a dependency-free PNG decode → 32×32 downscale → encode) prints the base64 to paste into `tray.ts`'s `ICON_PNG`, so dev and the packaged app load identical bytes. New logo → drop it in as `build/icon.png`, run `gen-tray.mjs`.
- **Startup splash** (`main/splash.ts`): a centered, frameless, transparent, always-on-top window showing `build/splash.png`, created first thing in `whenReady` and dismissed once the four startup data fetches (`statsReady`/`exchangeReady`/`baseTypesReady`/`leaguesReady`) settle — but with a `SPLASH_MIN_MS` (2.5s) floor so it never just flashes and a `SPLASH_MAX_MS` (15s) cap so a hung network can't trap it. It fades in/out via a CSS opacity transition on the image (not window opacity, which is unreliable on transparent windows); `createSplashWindow` returns a `Splash` whose idempotent `close()` fades out then destroys and whose `setStatus(text)` swaps the version label for an update status line. The image is bundled via electron-vite's `?asset` import (copied into `out/main/chunks`, packed into the asar — see the `*?asset` decl in `main/env.d.ts`) and inlined as a data URL, so unlike the tray it needs no base64 regen step. The other `build/*.png` are referenced the same way only by electron-builder, not at runtime. **The splash doubles as the startup-update narrator** (issue #65): `initAutoUpdater` takes an `onStartup(status)` callback; `index.ts` gates the splash's normal dismissal on **both** the data fetches settling (`dataSettled`) **and** the startup update check resolving (`updateResolved`, set on `available`/`downloading`/`downloaded`/`not-available`/`error`), so a check that lands after the fetches can't close the card before we pin it. An `available`/`downloading` status **pins the splash open** (`splashPinnedForUpdate`, bypassing the `MAX_MS` cap) and shows "Updating to vX.Y.Z… / Downloading… N%" through the relaunch; `error` releases the pin and lets the card dismiss (app keeps running on the current version). `updateResolved` starts **true in dev** (the updater is a no-op there and never reports). To eyeball the narration without a real feed, run `TRADEWIND_FAKE_UPDATE=1 npm run dev` — it walks the splash through available → downloading → restarting, then dismisses (no actual relaunch).
- **Auto-update** (`main/updater.ts`, `electron-builder.yml`): GitHub-releases feed (provider config → baked `app-update.yml`; no URL in code). **Discovery is automatic; the binary download is not** — a background download can spike ping mid-game, so `autoUpdater.autoDownload = false` and we call `downloadUpdate()` explicitly. The in-flight check's **`source`** (`startup | poll | manual`) decides what an `update-available` does: **startup** downloads immediately and installs-and-relaunches (`quitAndInstall(true, true)`) — the user isn't in a match yet — but **narrates it on the splash** via the `onStartup` callback so it's not a mystery restart (issue #65; see the splash bullet); **poll** (every 15 min) pushes an `available` status so the renderer's `UpdateToast` shows an **Update** button (`tw:download-update` → `downloadUpdate()`), and only that click downloads; **manual** drives the stateful themed dialog. Once a download finishes it's **always either startup or an explicit click = consent to restart**, so `update-downloaded` calls `quitAndInstall()` straight away — **no second restart prompt**. While a `manualDialog` is open, update events route to it (`setContent`) instead of the toast. Silent installs require the NSIS **`oneClick: true`** installer (no wizard, no UAC, per-user) — `oneClick: false` shows the assisted wizard even for updates. Note: detection lag after publishing is GitHub's CDN caching the release feed, not us; restarts don't bust it.
- **ToS posture:** clipboard + overlay only — no memory reading, no injection, one game action per keypress. All GGG/poe2scout calls go through the single rate-limit queue honoring `X-Rate-Limit-*`/`Retry-After`. Honest `User-Agent`. Requires **windowed-fullscreen** (overlays can't draw over exclusive fullscreen).

## Git

Work on a feature branch, push it, and open a PR against `main` — `main` is
protected, no direct pushes. End commit messages with the `Co-Authored-By`
trailer.

Releases are tag-driven (push a `vX.Y.Z` tag → `.github/workflows/release.yml`).
The **`/release` skill** (`.claude/skills/release/SKILL.md`, committed so every
collaborator has it) cuts a release in two phases around the PR-merge gate:
Phase A infers the next version from commits since the last tag, bumps
`package.json`, prepends the user-facing entry to `CHANGELOG.md`, and opens the
release PR; Phase B (after merge) tags, lets CI publish, then sets the GitHub
Release notes from the `CHANGELOG.md` entry. `CHANGELOG.md` is the single source
of truth for the polished notes.

## Researching game mechanics

When you (or a collaborator) need to research PoE2 mechanics, items, or crafting
data, the canonical sources are the **wiki** (`poewiki.net`, the
`poe2wiki:` namespace — e.g. `poewiki.net/wiki/poe2wiki:<Page>`) and
**`poe2db.tw`** (the data dump the generator scripts already scrape — see
`scripts/gen-essences.mjs`). Prefer the wiki for "how does X work" and poe2db
for the per-item mod/target tables. Both occasionally `403` automated fetches;
retry or fetch the equivalent page on the other site.
