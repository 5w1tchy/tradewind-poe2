**Golden rule: `src/core` stays pure** ŌĆö no Electron, no DOM, no `window`. It is
imported by both the main process and the renderer and must stay framework- and
runtime-agnostic. All side effects (network, clipboard, windows) live in `main/`.

## The core loop

1. Native keyboard hook sees `Ctrl+D` while PoE2 is focused; the keypress is swallowed.
2. `main/` synthesizes the game's advanced copy (`Ctrl+Alt+C`), polls the clipboard, restores it.
3. `core/parser` ŌåÆ `ParsedItem`; `core/stats-db` matcher maps mod lines to stat IDs; `core/query/prepare` builds a `PreparedQuery`.
4. Main sends `tw:item` (`ItemPayload`) to the renderer, which shows the popup and auto-searches (uniques/gems/white bases) or arms the Search button (rares). **Currency-exchange items** (anything with an `exchangeId`) skip the live search entirely: main attaches a `currency` quote (poe2scout aggregate) to the payload and the renderer shows the chart view ŌĆö see the exchange gotcha below.
5. Renderer calls `search()` ŌåÆ main runs `POST /search` + `GET /fetch` through the single rate-limit queue; `core/pricing` produces the estimate; results render.

## Conventions & gotchas

- **React, not Vue.** `PLAN.md` predates the migration and still says "Vue 3 + Tailwind" ŌĆö ignore that; the renderer is React 19 with **CSS Modules** (`*.module.css`). There is **no Tailwind**. Generic class names (`.name`, `.row`, `.tier`) collide across components, which is why styles are scoped modules; only the global theme primitives (`tw-label`, `tw-btn`, `tw-menu`) in `src/renderer/src/styles/theme.css` are plain global class strings.
- **No `<StrictMode>`.** The preload appends IPC listeners with no unsubscribe, so the dev double-invoke would deliver every item twice. `onItem`/`onHide` are subscribed exactly once in `App.tsx`'s mount effect ŌĆö keep it that way.
- **Mod-origin tags** (issue #54, `core/query/prepare.ts` `originFor` ŌåÆ `PreparedStatFilter.origin`, rendered by `originTag()` in `PriceCheck.tsx`, colors in `PriceCheck.module.css`). A stat row shows a short colored tag *after* the affix badge for a mod with special provenance: **F** fractured, **C** crafted, **D** desecrated, **E** enhanced (anoint/enchant `{ Enhancement }`), **CE** corruption (`{ Corruption Enhancement }`). Origin is derived from the `ParsedMod` flags (`crafted`/`desecrated`/`fractured`) plus generation; a `{ Corruption Enhancement }` header parses as **enhancement-generation with `mod.corrupted` set** (vs a plain anoint), which is what splits CE from E. Synthetic rows (summed `(total)`, resistance pseudos, the hybrid-single convenience rows) carry **no** origin ŌĆö the in-place hybrid node owns the tag. **Origin only survives an advanced/inventory copy:** a chat link copies a corruption-enhancement as a bare `(enchant)` line with no marker, so it degrades to a plain enchant row with no CE tag ŌĆö same "don't guess what the clipboard didn't give us" rule as tier reconstruction.
- **Two clipboard copy formats; only chat links need reconstruction** (`core/parser/parse.ts`, `core/mod-pool/`). An **inventory** copy is the *advanced* form: each mod carries a `{ Prefix Modifier "ŌĆ”" (Tier: N) }` header, so the parser reads affix + tier (and the `(min-max)` roll range) straight off the text ŌĆö exact, no guessing. A **chat-linked** item copies in the *basic* form: bare mod lines, **no headers, no tier, no ranges** (implicits are still tagged `(implicit)` and split out by the parser). For basic copies, `core/mod-pool/modPool.ts` *reconstructs* affix + tier from the rolled value against the bundled `pool.json` (slimmed repoe-fork mods) + `bases.json`: match the stat text, keep brackets the roll fits, read back `generation_type` + the tier index within that base-spawnable family. **`prepare.ts` only ever reconstructs when `mod.generation === 'explicit' && tier === null`** ŌĆö advanced/inventory mods keep their parsed header values untouched. **Crafted/desecrated mods reuse the regular tier ladder** (a `(crafted)` 26% cast speed is the same T3 a rolled one would be), so they reconstruct like any explicit ŌĆö don't special-case them out. Correctness rules in the matcher: spawn weights are **ordered, first-matching-tag-wins** (`!`-prefixed tag = weight 0; a naive "any positive tag" test over-counts tiers and is wrong); brackets come from the **display text** not `stats` (leech is stored permyriad, "reduced" is stored negative); a roll whose tiers overlap, or two mods that collide on one **group** (impossible on a real item), yields no tier; and when a tier can't be pinned but the stat has one fixed affix slot, `affixForStat` still badges the **affix only** (P/S, no number). A roll that fits **both** a prefix and a suffix bracket (e.g. Rarity of Items found) is resolved by **affix-slot constraint** ŌĆö once the confident mods fill one side (rares cap at 3+3, magic at 1+1) the ambiguous one is forced to the open side (`reconstructExplicits` in `prepare.ts`); the rare `+1 prefix/suffix` crafts can exceed the cap, a documented edge. A wrong number never shows ŌĆö a blank or affix-only beats a guess. ~99% never-wrong on gear; residual blanks are genuine value ambiguities or stats absent from the upstream dump (e.g. the maximum-mana *suffix* family). Re-run `node scripts/gen-mod-pool.mjs` (reads `scratch/mods.min.json` + `base_items.min.json`, else fetches repoe-fork) after PoE2 patches. **Source is repoe-fork, not poe2db** ŌĆö repoe-fork is structured JSON with the tiers/tags we need; poe2db has no API (HTML only, scraped solely for essences).
- **IPC channels** (`tw:*`) are defined in `preload/index.ts` and handled in `main/index.ts`; the typed surface is `src/shared/ipc.ts`. Add new calls in all three.
- **Essence advice respects the crafted-mod cap** (issue #24, `core/craft/craftedSlots.ts`, `essencesForItem`, `CraftPane.tsx`). A Perfect or corrupted essence augments a Rare with a guaranteed mod in its **single crafted slot**, so on a Rare that already fills that slot the game blocks it. `craftedSlots(item)` counts `explicits.filter(m => m.crafted)` (a mod that is **both fractured and crafted** still occupies the slot ŌĆö this is why `modHeader`'s descriptor regex accepts a *stacked* variant prefix like `Fractured Crafted`, not a single keyword) and reads the cap off the rune lines: base 1, **+N per `Can have N additional Crafted Modifier` rune** (Astrid's Creativity, a socketed Soul Core, copies as that line). Main attaches `{ used, cap }` to `ItemPayload.craftedSlots`; `essencesForItem(class, rarity, craftedSlots?)` drops Perfect **and** corrupted essences when `used >= cap`, replacing the list with a plain reason note instead of silently hiding them. Only Rare gear is gated (Magic/Normal don't have the slot yet). The shared `craftedCapNote(crafted, blocked)` (in `craftedSlots.ts`) builds the reason line for both essences and **liquids** (see next). **Each guaranteed mod in `essences.json` also carries `groups` + `affix`** (joined from the repoe-fork `mods.min.json` by `gen-essences.mjs` ŌĆö match poe2db's display text to a gear mod by an order-independent word multiset, disambiguating by the granted mod's **exact roll bracket first** (the range pins the tier, hence the prefix/suffix *family* for a stat that rolls as both ŌĆö e.g. a (15-18)% Rarity essence is the suffix `ItemFoundRarityIncrease`, since the prefix family's neighbouring tier is (16-19)), then by the non-rollable candidate; ambiguous text yields the union of groups, special mods like "Mark of the Abyssal Lord" yield none). This is the data layer for the **mod-group-conflict gate**: two mods can't share a group, so a guaranteed-mod craft is blocked when the item already has a mod in the guaranteed mod's group (e.g. Celestial Alloy's `IncreasedMana` mod on a wand that already has max Mana).
- **Essence advice flags mod-group conflicts** (issue #72, `core/craft/conflict.ts`, `core/mod-pool/modPool.ts` `groupsForLine`, `essencesForItem`, `CraftPane.tsx`). Built on the `groups` data above. Two mods can never share a group, so an essence's guaranteed mod is **refused by the game when the item already carries a mod in that group** ŌĆö both for a Perfect/corrupted essence augmenting a Rare *and* a Greater essence upgrading a Magic item (e.g. Greater Essence of Opulence on a Magic item that already has a Rarity suffix). Wiring: `collectItemMods(parsed, baseType)` (the `itemMods` export of `conflict.ts`) pairs each **explicit** mod's line text with its repoe group(s) via `groupsForLine` (a deliberately loose pool lookup ŌĆö all candidate groups for the stat text, base-narrowed when known, since a group is intrinsic to the stat and base-independent; empty when the stat isn't in `pool.json`, so it blocks nothing). `groupsForLine` takes the mod's **affix** (from the advanced copy's prefix/suffix header) and filters candidates by it, so a stat that rolls as both a prefix and a suffix under *different* groups (Rarity of Items found) resolves to the right family ŌĆö a suffix-rarity item mod doesn't cross-block a suffix-rarity essence's prefix-rarity twin, and vice versa. A basic chat-link copy carries no affix, so both groups stay (the conservative "don't guess" default). This is general, not a rarity special-case ŌĆö many essences grant such affix-split mods. Main attaches that to `ItemPayload.itemMods`; `essencesForItem(class, rarity, craftedSlots?, itemMods?)` sets `EssenceForItem.blockedBy` to the conflicting mod's display text via `conflictingMod(modGroups, itemMods)`. Unlike the #24 cap gate, **blocked essences stay listed** (not hidden) ŌĆö the renderer greys the row and shows `blocked by "<mod>"` on the right, so the user sees *why*. The gate no-ops without `itemMods` (every essence lists unblocked). The same `conflict.ts` helpers are the shared check the **#51 Alloys** advisor will reuse. Preview: the default Rings item ships `SAMPLE_ITEM_MODS` (a `MaximumManaIncreasePercent` mod) so Perfect Essence of the Mind shows blocked ŌĆö use `?used=0` to clear the #24 cap block and see the gate on the list.
- **Liquid Emotions are the jewel-crafting sibling of essences** (issue #48, `core/craft/liquids.ts`, `liquids.json` from `scripts/gen-liquids.mjs`, the **Liquids** sub-tab in `CraftPane.tsx`). Applied to a **Rare jewel**, a liquid removes a random modifier and augments it with a guaranteed **crafted** mod ŌĆö mechanically the corrupted-essence `rareOnly` path, so it reuses the **#24 crafted-cap gating** (a liquid-applied mod copies as a real `{ Crafted ŌĆ” Modifier }`, so `craftedSlots()` already counts it; jewels have no rune lines so the cap stays 1). **The key difference from essences: liquids key off the jewel's *gem base type*, not the item class** ŌĆö every jewel is `Item Class: Jewels`, so `liquidsForItem(itemClass, baseType, rarity, crafted?)` reads the base type (rares carry it on `prepared.baseTypeFilter.value`, uniques on `prepared.type`): `Time-Lost <gem>` ŌåÆ Time-Lost jewel, bare `Ruby/Sapphire/Emerald/Diamond` ŌåÆ basic jewel; an unknown base (or non-jewel class) returns an empty no-note result. The poe2db table is **gem├Śaffix keyed** (`Ruby Prefix: ŌĆ”`, `Emerald Suffix: ŌĆ”`) and the **target family is read from the behavior sentence** (`ŌĆ”Augments a Rare Basic JewelŌĆ”` vs `ŌĆ”Time-Lost JewelŌĆ”`), so each liquid stores `mods: [{ gem, affix, text }]`. **Ruby/Sapphire/Emerald are attribute jewels** ŌĆö each takes only its own column (one deterministic mod). **Diamond is the wildcard jewel**: if the liquid has its own **Diamond** column it gives only that (e.g. Isolation ŌåÆ chaos res only); lacking one, Diamond rolls a *random one of* the attribute variants (e.g. Ire ŌåÆ armour/ES/evasion). So `liquidsForItem` returns the Diamond column when present else every line for Diamond (deduped by affix+text), and only the matching-gem line(s) for the attribute gems. Since a liquid always adds exactly one mod, **`LiquidForItem.mods` is the set of possible outcomes** ŌĆö >1 entry means "rolls one of" (the renderer tags it), which covers both Diamond and the **Potent** liquids (Ferocity/Contempt/Melancholy) that offer a prefix *or* suffix. Potent liquids are gold-tinted vs base-emotion bronze and sort last. Liquids only apply to **Rare** jewels (they remove-and-augment, they don't upgrade rarity like Greater essences). Re-run `node scripts/gen-liquids.mjs` after PoE2 patches ŌĆö it scrapes poe2db's `#LiquidItem` pane (the page's other tabs are the out-of-scope 3-liquid amulet-instilling passives) and downloads icons to `assets/liquids/`. Preview with `?class=Jewels&base=Emerald&rarity=Rare&used=0` (`preview.ts` `base` param).
- **Currency-exchange items price off poe2scout, not the live trade API** (`main/exchangeSnapshot.ts`, `core/exchange/`, `renderer/components/CurrencyView.tsx` + `PriceChart.tsx`). Everything the in-game Currency Exchange (Ange) trades ŌĆö currency, fragments, runes, essences, lineage support gems, ŌĆ” ŌĆö carries **one aggregate price**, not a per-roll order book, so a live search is the wrong tool. When `prepareQuery` sets an `exchangeId`, `priceCheck` resolves a `currency` quote from poe2scout's **flat `/{realm}/Leagues/{league}/Items`** snapshot (one ~450KB GET covering every exchange item, cached via `cachedFetchJson` with a short `freshMs` since prices move) and attaches it to `ItemPayload`. The join is exact: **GGG static exchange id == poe2scout `ApiId` == `exchangeId`** (e.g. `rakiatas-flow`). The renderer branches in `PriceCheck` on `payload.currency` to render `CurrencyView` instead of the search/listings UI ŌĆö **no Search button, no pricing controls, no API call** ŌĆö showing the price in all three denominations (ex/div/chaos, derived from the snapshot's own `divine`/`chaos` rows) plus a hand-rolled SVG **price + volume chart**. Chart history (`PriceLogs`, 6-hourly) is fetched lazily per item via `tw:currency-history`; switching league re-resolves via `tw:currency-quote`. **It's a soft dependency** ŌĆö a snapshot miss/failure leaves `currency` null and the item falls back to the existing live exchange path, so poe2scout being down degrades, never breaks. (This is the same source as the optional `scoutAnchor` cross-check, here promoted to primary for exchange items only ŌĆö uniques/rares still price live.)
- **Uniques get an instant poe2scout aggregate banner + a manual live search** (issue #80, `main/exchangeSnapshot.ts` `uniqueQuote`, `core/exchange` `UniqueQuote`, `renderer/components/UniqueQuoteBanner.tsx`). The same flat `/Items` snapshot that prices currency **also carries every unique** (466 of them, each with `Name`/`Type`/`CurrentPrice`/`IconUrl`/`ItemId` but **`ApiId: null`**), so a rough ballpark is free at check time. The provider builds a **second index keyed by `name|type`** (uniques have no ApiId) alongside `byApiId`; `uniqueQuote(league, name, type)` joins on parsed `prepared.name` + `prepared.type` (normalized case/space-insensitively). **Unlike currency, this is a soft *anchor*, not a replacement** ŌĆö poe2scout's aggregate is noisy for cheap/thinly-traded uniques (a 1-ex leveling unique can report ~97k ex at quantity 1ŌĆō3), so a unique still prices **live, per-roll**: the banner is the instant ballpark, the search below is the truth. Two behavior changes for `rarity === 'Unique'`, both in `PriceCheck.tsx`: (1) it's **excluded from the auto-search gate** (the `name || type || exchangeId` branch) so uniques **arm the Search button** like rares instead of auto-firing a rate-limited search on every hover ŌĆö the whole point is cutting GGG `search` traffic; (2) `UniqueQuoteBanner` renders on top of the Price tab ŌĆö the price in ex/div/chaos with currency-orb icons (from `payload.currencyIcons`, same source as `CurrencyView`) + the item art + an "<rarity> ŌŚå poe2scout aggregate" eyebrow. It's styled as a *sliver of the item's own rarity tooltip*: a single inline `--accent` CSS var (set from the rarity color, `RARITY_ACCENT`) is `color-mix`ed through the spine/wash/border/label so the banner reads as "unique" at a glance, while the price number itself stays **gold** (value is gold ŌĆö the overlay-wide rule). **Non-blocking by design:** the banner requests its quote on mount via the new `tw:unique-quote` IPC (`getUniqueQuote`, mirrors `getCurrencyQuote`) ŌĆö main's `priceCheck` hot path is **unchanged** (no quote attached to `ItemPayload`), so the popup is instant even on a cold snapshot. A miss/failure ŌåÆ no banner, Search still works (same soft-dependency posture as currency). Preview with `?rarity=Unique` (`preview.ts` seeds `SAMPLE_UNIQUE`). Out of scope (follow-up): a lazy price-history sparkline via `/Items/{ItemId}/History` and a `Quantity`-based liquidity hint.
- **Popup is a fixed-size, resizable box** (`App.tsx`, issue #35): the popup `<div>` has an explicit width/height (not content-sized), placed **top-middle** (horizontally centered, ~8% down) and **draggable** from the header bar. Because the frame size is fixed, **content changes never resize it** ŌĆö switching PriceŌåöCraft or loading results leaves the frame put; each tab panel fills the frame via the flex-column `.body` and **scrolls internally** (the stats list, the results list, the essence list own their own overflow). Drag the **bottom-right corner handle** to resize the whole popup; on the Price tab, drag the **grip above the results list** to set its height while the stats list flex-fills the rest. Both sizes persist to `config.json` (`popupSize` via `tw:set-popup-size`, `resultsHeight` via `tw:set-results-height`) and are restored on the next check (sent back in `ItemPayload`). Anchor resize gestures to the rendered `size`/`resultsHeight` state, **not** `el.offsetWidth` (which includes padding+border and snaps the first drag tick). `config.ts`'s `sanitize()` coerces/clamps every loaded field, so a malformed or schema-changed `config.json` heals to defaults instead of leaking `NaN`. The **buyout-price currency** is persisted the same way (issue #20): picking one in the buyout popover fires `setBuyoutCurrency` ŌåÆ `tw:set-buyout-currency`, main stores it as `config.buyoutCurrency`, and `prepareQuery` seeds it into every new check's `buyout.option` (via `PrepareOptions.buyoutOption`) so the picker remembers the choice ŌĆö `null` = Exalted Orb Equivalent, the only value that adds no price filter; any other option alone restricts listings to that currency.
- **Overlay popup behavior** (`App.tsx` + `main/index.ts`): the popup mimics PoE2's item tooltip (issue #32). The top-right header button is one **morphing control** (`setPinned` ŌåÆ `tw:set-pinned`): a **pin** glyph while unpinned (click to pin), an **Ō£Ģ** while pinned (click to close ŌĆö there is no separate unpin). **Unpinned** (default): an outside click or Esc closes it. **Pinned**: outside clicks are ignored; only the Ō£Ģ or Esc close it, and a **fresh price check updates the popup in place and keeps the pin** ŌĆö pin state persists across searches and resets to unpinned **only when the popup closes** (`hidePopup` for Esc/outside; the null `tw:popup-rect` for the Ō£Ģ). The renderer reports its rect via `setPopupRect`; the main process hit-tests the cursor against it to toggle click-through (`setIgnoreMouseEvents`) and, on a `mousedown` outside it (observed via uiohook), auto-hides when unpinned. Report the rect synchronously from placement; never from a stale `pos` state (a `{0,0}` ghost report makes main treat the popup as off in the corner).
- **Software rendering on purpose.** `main/index.ts` calls `app.disableHardwareAcceleration()` at startup. On machines where the GPU is present but only half-engaged for Chromium (it shows up yet `app.getGPUFeatureStatus().gpu_compositing` is a `disabled_software`/WARP fallback), letting Electron try the GPU and limp along in that path makes the popup lag hard (jerky drag, multi-second hover repaints). The overlay is a tiny, mostly static UI that doesn't need the GPU, so forcing clean software rendering is smooth everywhere and dodges that whole fallback class. Don't "optimize" this away.

The native tier prevents `Ctrl+D` leaking as a WASD-style game input.

- **Startup splash** (`main/splash.ts`): a centered, frameless, transparent, always-on-top window showing `build/splash.png`, created first thing in `whenReady` and dismissed once the four startup data fetches (`statsReady`/`exchangeReady`/`baseTypesReady`/`leaguesReady`) settle ŌĆö but with a `SPLASH_MIN_MS` (2.5s) floor so it never just flashes and a `SPLASH_MAX_MS` (15s) cap so a hung network can't trap it. The image is bundled via electron-vite's `?asset` import (copied into `out/main/chunks`, packed into the asar ŌĆö see the `*?asset` decl in `main/env.d.ts`) and inlined as a data URL, so unlike the tray it needs no base64 regen step. The other `build/*.png` are referenced the same way only by electron-builder, not at runtime. **The splash doubles as the startup-update narrator** (issue #65): `initAutoUpdater` takes an `onStartup(status)` callback; `index.ts` gates the splash's normal dismissal on **both** the data fetches settling (`dataSettled`) **and** the startup update check resolving (`updateResolved`, set on `available`/`downloading`/`downloaded`/`not-available`/`error`), so a check that lands after the fetches can't close the card before we pin it. An `available`/`downloading` status **pins the splash open** (`splashPinnedForUpdate`, bypassing the `MAX_MS` cap) and shows "Updating to vX.Y.ZŌĆ” / DownloadingŌĆ” N%" through the relaunch; `error` releases the pin and lets the card dismiss (app keeps running on the current version). `updateResolved` starts **true in dev** (the updater is a no-op there and never reports). To eyeball the narration without a real feed, run `TRADEWIND_FAKE_UPDATE=1 npm run dev` ŌĆö it walks the splash through available ŌåÆ downloading ŌåÆ restarting, then dismisses (no actual relaunch).
- **Auto-update** (`main/updater.ts`, `electron-builder.yml`): GitHub-releases feed (provider config ŌåÆ baked `app-update.yml`; no URL in code). **Discovery is automatic; the binary download is not** ŌĆö a background download can spike ping mid-game, so `autoUpdater.autoDownload = false` and we call `downloadUpdate()` explicitly. The in-flight check's **`source`** (`startup | poll | manual`) decides what an `update-available` does: **startup** downloads immediately and installs-and-relaunches (`quitAndInstall(true, true)`) ŌĆö the user isn't in a match yet ŌĆö but **narrates it on the splash** via the `onStartup` callback so it's not a mystery restart (issue #65; see the splash bullet); **poll** (every 15 min) pushes an `available` status so the renderer's `UpdateToast` shows an **Update** button (`tw:download-update` ŌåÆ `downloadUpdate()`), and only that click downloads; **manual** drives the stateful themed dialog. Once a download finishes it's **always either startup or an explicit click = consent to restart**, so `update-downloaded` calls `quitAndInstall()` straight away ŌĆö **no second restart prompt**. While a `manualDialog` is open, update events route to it (`setContent`) instead of the toast. Silent installs require the NSIS **`oneClick: true`** installer (no wizard, no UAC, per-user) ŌĆö `oneClick: false` shows the assisted wizard even for updates. Note: detection lag after publishing is GitHub's CDN caching the release feed, not us; restarts don't bust it.
- **ToS posture:** clipboard + overlay only ŌĆö no memory reading, no injection, one game action per keypress. All GGG/poe2scout calls go through the single rate-limit queue honoring `X-Rate-Limit-*`/`Retry-After`. Honest `User-Agent`. Requires **windowed-fullscreen** (overlays can't draw over exclusive fullscreen).

## Git

Work on a feature branch, push it, and open a PR against `main` ŌĆö `main` is
protected, no direct pushes. End commit messages with the `Co-Authored-By`
trailer.

Releases are tag-driven (push a `vX.Y.Z` tag ŌåÆ `.github/workflows/release.yml`).
The **`/release` skill** (`.claude/skills/release/SKILL.md`, committed so every
collaborator has it) cuts a release in two phases around the PR-merge gate:
Phase A infers the next version from commits since the last tag, bumps
`package.json`, prepends the user-facing entry to `CHANGELOG.md`, and opens the
release PR; Phase B (after merge) tags, lets CI publish, then sets the GitHub
Release notes from the `CHANGELOG.md` entry. `CHANGELOG.md` is the single source
of truth for the polished notes.

## Researching game mechanics

When you (or a collaborator) need to research PoE2 mechanics, items, or crafting
data, the canonical sources are the **wiki** (`poewiki.net`, the
`poe2wiki:` namespace ŌĆö e.g. `poewiki.net/wiki/poe2wiki:<Page>`) and
**`poe2db.tw`** (the data dump the generator scripts already scrape ŌĆö see
`scripts/gen-essences.mjs`). Prefer the wiki for "how does X work" and poe2db
for the per-item mod/target tables. Both occasionally `403` automated fetches;
retry or fetch the equivalent page on the other site.
