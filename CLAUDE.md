# Tradewind ‚Äî PoE2 price-check overlay

One hotkey on a hovered Path of Exile 2 item ‚Üí an overlay popup with a price
estimate and live trade listings. Electron + TypeScript + React.

> overlay/popup behavior, the input tiers, or any convention/gotcha below ‚Äî

## Commands

```
npm run dev        # electron-vite dev ‚Äî launches the overlay, watches for the PoE2 window, HMR
npm run build      # electron-vite build ‚Üí out/
npm start          # electron-vite preview (built app)
npm test           # vitest run ‚Äî parser/matcher/query/pricing/craft unit tests
npm run fixtures   # re-split fixtures/inbox.txt into per-item test fixtures
npx tsc -b         # typecheck (project refs: tsconfig.node.json + tsconfig.web.json)
```

Browser UI preview without the game or Electron:
`npx vite --config vite.preview.config.ts` then open `/preview.html` (mocks
`window.tradewind` with a canned item ‚Äî see `src/renderer/src/preview.ts`).

## Layout & the process boundary

```
src/
‚îú‚îÄ core/      Pure TypeScript, NO electron imports. Unit-tested. The brain.
‚îÇ   ‚îú‚îÄ parser/    clipboard item text ‚Üí ParsedItem (splits on "--------" sections)
‚îÇ   ‚îú‚îÄ stats-db/  live-fetched /api/trade2/data ‚Üí runtime matcher generation (# placeholders ‚Üí regex)
‚îÇ   ‚îú‚îÄ query/     ParsedItem + filters ‚Üí PreparedQuery ‚Üí trade2 search body (prepare.ts, build.ts)
‚îÇ   ‚îú‚îÄ pricing/   currency normalization, outlier rejection, estimate range + confidence
‚îÇ   ‚îú‚îÄ craft/     essence crafting helper (essences.json generated by scripts/gen-essences.mjs)
‚îÇ   ‚îî‚îÄ trade/     SearchOutcome / listing types
‚îú‚îÄ main/      Electron main process. Win32 window tracking, global hotkey, clipboard, API gateway.
‚îú‚îÄ preload/   contextBridge ‚Äî exposes window.tradewind (see src/shared/ipc.ts for the surface)
‚îú‚îÄ renderer/  React 19 (.tsx + CSS Modules). The popup UI.
‚îî‚îÄ shared/    ipc.ts ‚Äî the typed IPC contract shared by main, preload, renderer
```

**Golden rule: `src/core` stays pure** ‚Äî no Electron, no DOM, no `window`. It is
imported by both the main process and the renderer and must stay framework- and
runtime-agnostic. All side effects (network, clipboard, windows) live in `main/`.

## The core loop

1. Native keyboard hook sees `Ctrl+D` while PoE2 is focused; the keypress is swallowed.
2. `main/` synthesizes the game's advanced copy (`Ctrl+Alt+C`), polls the clipboard, restores it.
3. `core/parser` ‚Üí `ParsedItem`; `core/stats-db` matcher maps mod lines to stat IDs; `core/query/prepare` builds a `PreparedQuery`.
4. Main sends `tw:item` (`ItemPayload`) to the renderer, which shows the popup and auto-searches (uniques/gems/currency/white bases) or arms the Search button (rares).
5. Renderer calls `search()` ‚Üí main runs `POST /search` + `GET /fetch` through the single rate-limit queue; `core/pricing` produces the estimate; results render.

## Conventions & gotchas

- **React, not Vue.** `PLAN.md` predates the migration and still says "Vue 3 + Tailwind" ‚Äî ignore that; the renderer is React 19 with **CSS Modules** (`*.module.css`). There is **no Tailwind**. Generic class names (`.name`, `.row`, `.tier`) collide across components, which is why styles are scoped modules; only the global theme primitives (`tw-label`, `tw-btn`, `tw-menu`) in `src/renderer/src/styles/theme.css` are plain global class strings.
- **No `<StrictMode>`.** The preload appends IPC listeners with no unsubscribe, so the dev double-invoke would deliver every item twice. `onItem`/`onHide` are subscribed exactly once in `App.tsx`'s mount effect ‚Äî keep it that way.
- **IPC channels** (`tw:*`) are defined in `preload/index.ts` and handled in `main/index.ts`; the typed surface is `src/shared/ipc.ts`. Add new calls in all three.
- **Popup is a fixed-size, resizable box** (`App.tsx`, issue #35): the popup `<div>` has an explicit width/height (not content-sized), placed **top-middle** (horizontally centered, ~8% down) and **draggable** from the header bar. Because the frame size is fixed, **content changes never resize it** ‚Äî switching Price‚ÜîCraft or loading results leaves the frame put; each tab panel fills the frame via the flex-column `.body` and **scrolls internally** (the stats list, the results list, the essence list own their own overflow). Drag the **bottom-right corner handle** to resize the whole popup; on the Price tab, drag the **grip above the results list** to set its height while the stats list flex-fills the rest. Both sizes persist to `config.json` (`popupSize` via `tw:set-popup-size`, `resultsHeight` via `tw:set-results-height`) and are restored on the next check (sent back in `ItemPayload`). Anchor resize gestures to the rendered `size`/`resultsHeight` state, **not** `el.offsetWidth` (which includes padding+border and snaps the first drag tick). `config.ts`'s `sanitize()` coerces/clamps every loaded field, so a malformed or schema-changed `config.json` heals to defaults instead of leaking `NaN`.
- **Overlay popup behavior** (`App.tsx` + `main/index.ts`): the popup mimics PoE2's item tooltip (issue #32). The top-right header button is one **morphing control** (`setPinned` ‚Üí `tw:set-pinned`): a **pin** glyph while unpinned (click to pin), an **‚úï** while pinned (click to close ‚Äî there is no separate unpin). **Unpinned** (default): an outside click or Esc closes it. **Pinned**: outside clicks are ignored; only the ‚úï or Esc close it, and a **fresh price check updates the popup in place and keeps the pin** ‚Äî pin state persists across searches and resets to unpinned **only when the popup closes** (`hidePopup` for Esc/outside; the null `tw:popup-rect` for the ‚úï). The renderer reports its rect via `setPopupRect`; the main process hit-tests the cursor against it to toggle click-through (`setIgnoreMouseEvents`) and, on a `mousedown` outside it (observed via uiohook), auto-hides when unpinned. Report the rect synchronously from placement; never from a stale `pos` state (a `{0,0}` ghost report makes main treat the popup as off in the corner).
- **Software rendering on purpose.** `main/index.ts` calls `app.disableHardwareAcceleration()` at startup. On machines where the GPU is present but only half-engaged for Chromium (it shows up yet `app.getGPUFeatureStatus().gpu_compositing` is a `disabled_software`/WARP fallback), letting Electron try the GPU and limp along in that path makes the popup lag hard (jerky drag, multi-second hover repaints). The overlay is a tiny, mostly static UI that doesn't need the GPU, so forcing clean software rendering is smooth everywhere and dodges that whole fallback class. Don't "optimize" this away.
- **Input has three tiers** (`main/index.ts`, `keyhook.ts`, `input.ts`): native `WH_KEYBOARD_LL` addon (`tradewind-keyhook` via koffi, best ‚Äî blocks the key before the game's key-state table) ‚Üí `globalShortcut` ‚Üí `uiohook-napi` observe-only. The native tier prevents `Ctrl+D` leaking as a WASD-style game input.
- **Tests are fixture-driven.** `fixtures/` holds real in-game `Ctrl+Alt+C` copies. ‚öÝÔ∏è `npm run fixtures` (`scripts/split-fixtures.mjs`) **wipes any manual fixture not present in `fixtures/inbox.txt`** ‚Äî add new samples to `inbox.txt`, don't hand-place fixture files.
- **Tray is the only chrome.** The sole window is the click-through overlay, hidden whenever PoE2 isn't focused ‚Äî so the system-tray icon (`main/tray.ts`) is the only way to quit (Quit) and to trigger a manual update check (Check for updates‚Ä¶, wired to `checkForUpdatesManually` in `updater.ts`, which opens a **themed dialog**, not the native grey box: `main/themed-dialog.ts`'s `showThemedDialog` draws the same kind of frameless/transparent/always-on-top window as the tray menu (centered, draggable card, `tw://btn/<i>` nav channel) but is **stateful** ‚Äî it returns a `DialogController` whose `setContent()` morphs the live window in place, so the manual check walks one window through Checking‚Ä¶ ‚Üí Available (+ Update button) ‚Üí Downloading‚Ä¶ ‚Üí relaunch. Buttons are callback-based (`onClick`/`keepOpen`); Esc/Enter map to the `cancel`/`primary` buttons). Its menu is **not** the native Win32 menu (`tray.setContextMenu`), which can't be themed: `main/tray-menu.ts` draws a **custom themed popup** (`showTrayMenu`) matching the overlay ‚Äî a frameless/transparent/always-on-top `BrowserWindow` (same recipe as `splash.ts`, inline-HTML data URL, palette mirrored from `theme.css`), shown at the cursor on **either** tray click (left included, since the native menu's right-click-only is unintuitive for a one-icon app). It owns no preload/IPC: rows navigate to a `tw://item/<i>` scheme that `will-navigate` intercepts; Esc or blur (click-away) dismisses. Keep the `tray` reference alive in `index.ts` or the GC reaps it from the notification area. The app/installer icon is `build/icon.png` (the compass logo, already cropped tight to its content), picked up automatically by electron-builder. `build/` isn't bundled into the asar, so the tray can't read it at runtime ‚Äî `scripts/gen-tray.mjs` (a dependency-free PNG decode ‚Üí 32√ó32 downscale ‚Üí encode) prints the base64 to paste into `tray.ts`'s `ICON_PNG`, so dev and the packaged app load identical bytes. New logo ‚Üí drop it in as `build/icon.png`, run `gen-tray.mjs`.
- **Startup splash** (`main/splash.ts`): a centered, frameless, transparent, always-on-top window showing `build/splash.png`, created first thing in `whenReady` and dismissed once the four startup data fetches (`statsReady`/`exchangeReady`/`baseTypesReady`/`leaguesReady`) settle ‚Äî but with a `SPLASH_MIN_MS` (2.5s) floor so it never just flashes and a `SPLASH_MAX_MS` (10s) cap so a hung network can't trap it. It fades in/out via a CSS opacity transition on the image (not window opacity, which is unreliable on transparent windows); `createSplashWindow` returns a `Splash` whose idempotent `close()` fades out then destroys. The image is bundled via electron-vite's `?asset` import (copied into `out/main/chunks`, packed into the asar ‚Äî see the `*?asset` decl in `main/env.d.ts`) and inlined as a data URL, so unlike the tray it needs no base64 regen step. The other `build/*.png` are referenced the same way only by electron-builder, not at runtime.
- **Auto-update** (`main/updater.ts`, `electron-builder.yml`): GitHub-releases feed (provider config ‚Üí baked `app-update.yml`; no URL in code). **Discovery is automatic; the binary download is not** ‚Äî a background download can spike ping mid-game, so `autoUpdater.autoDownload = false` and we call `downloadUpdate()` explicitly. The in-flight check's **`source`** (`startup | poll | manual`) decides what an `update-available` does: **startup** downloads immediately and silently installs-and-relaunches (`quitAndInstall(true, true)`) ‚Äî the user isn't in a match yet; **poll** (every 15 min) pushes an `available` status so the renderer's `UpdateToast` shows an **Update** button (`tw:download-update` ‚Üí `downloadUpdate()`), and only that click downloads; **manual** drives the stateful themed dialog. Once a download finishes it's **always either startup or an explicit click = consent to restart**, so `update-downloaded` calls `quitAndInstall()` straight away ‚Äî **no second restart prompt**. While a `manualDialog` is open, update events route to it (`setContent`) instead of the toast. Silent installs require the NSIS **`oneClick: true`** installer (no wizard, no UAC, per-user) ‚Äî `oneClick: false` shows the assisted wizard even for updates. Note: detection lag after publishing is GitHub's CDN caching the release feed, not us; restarts don't bust it.
- **ToS posture:** clipboard + overlay only ‚Äî no memory reading, no injection, one game action per keypress. All GGG/poe2scout calls go through the single rate-limit queue honoring `X-Rate-Limit-*`/`Retry-After`. Honest `User-Agent`. Requires **windowed-fullscreen** (overlays can't draw over exclusive fullscreen).

## Git

Work on a feature branch, push it, and open a PR against `main` ‚Äî `main` is
protected, no direct pushes. End commit messages with the `Co-Authored-By`
trailer.

Releases are tag-driven (push a `vX.Y.Z` tag ‚Üí `.github/workflows/release.yml`).
The **`/release` skill** (`.claude/skills/release/SKILL.md`, committed so every
collaborator has it) cuts a release in two phases around the PR-merge gate:
Phase A infers the next version from commits since the last tag, bumps
`package.json`, prepends the user-facing entry to `CHANGELOG.md`, and opens the
release PR; Phase B (after merge) tags, lets CI publish, then sets the GitHub
Release notes from the `CHANGELOG.md` entry. `CHANGELOG.md` is the single source
of truth for the polished notes.
